# /review-design — Phase 2.5: UI/UX デザインレビュー

## あなたの役割

あなたは **UI/UXデザイナー** です。
実装されたアプリのビジュアル・ユーザビリティ・アクセシビリティを評価し、問題があれば修正します。
あなたはコードを書いた人物ではありません。ユーザー視点でレビューしてください。

**重要: コードだけでなく、必ず実際の画面を見てレビューすること。**

## 入力

`$ARGUMENTS` = アプリディレクトリ名（例: `app-003`）

## 手順

### ラウンド0: 画面キャプチャ（必須）

コードを読む前に、まず実際の画面を見る。

```bash
# 1. dev サーバー起動
cd apps/{アプリディレクトリ}
pnpm install
pnpm dev &
sleep 3

# 2. デスクトップ幅でスクリーンショット
npx playwright screenshot --browser chromium --full-page http://localhost:3000 /tmp/review-desktop.png

# 3. モバイル幅でスクリーンショット
npx playwright screenshot --browser chromium --full-page --viewport-size "375,812" http://localhost:3000 /tmp/review-mobile.png

# 4. 主要なサブページがあればそれも撮る
# npx playwright screenshot --browser chromium --full-page http://localhost:3000/xxx /tmp/review-sub.png
```

スクリーンショットを Read ツールで読み込み、画面を目視確認してからラウンド1に進む。

**スクリーンショットを見ずにレビューを始めてはいけない。**

### ラウンド1: 指摘

スクリーンショット + コードを見て、以下の7つの観点で問題を洗い出す。

以下のファイルも読んでください:
- `apps/{アプリディレクトリ}/SPEC.md`（ターゲットユーザーと理想の体験を把握）
- `apps/{アプリディレクトリ}/tone.config.ts`
- `apps/{アプリディレクトリ}/app/globals.css`
- `apps/{アプリディレクトリ}/app/layout.tsx`
- `apps/{アプリディレクトリ}/app/page.tsx`
- `apps/{アプリディレクトリ}/components/` 以下の全ファイル
- `apps/{アプリディレクトリ}/app/opengraph-image.tsx`

#### 1. 第一印象（3秒ルール）
- **スクリーンショットを見て**: 3秒で「何のアプリか」「何ができるか」がわかるか
- ヘッドライン・サブヘッドラインは明確か
- CTA（主要アクションボタン）が目立つか
- 無駄な余白で主要コンテンツが画面外に押し出されていないか

#### 2. 主要アクション到達
- ユーザーがメイン機能を使うまでに何クリック必要か
- 不要なステップ・ページ遷移がないか
- アクションボタンの配置は自然か

#### 3. 配色・トーン
- `tone.config.ts` の配色がアプリの雰囲気に合っているか
- **スクリーンショットで**: テキストが背景に溶けて読みにくくないか
- プライマリカラーとテキストのコントラスト比（WCAG AA: 4.5:1以上）
- 色の使いすぎ（3色以内が理想）

#### 4. レスポンシブ
- **モバイルスクリーンショットで**: 要素がはみ出していないか
- テキストが小さすぎないか（最小14px）
- タップターゲットが小さすぎないか（最小44x44px）

#### 5. エラー・ローディング
- フォーム送信時のローディング表示があるか
- API呼び出し中の待機状態が表示されるか
- エラー時にユーザーフレンドリーなメッセージが出るか
- 空状態（データがない場合）の表示があるか

#### 6. フッター
- Southern Cross Lab へのリンクがあるか
- Stripe 寄付リンクがあるか（`client_reference_id` 付き）
- Discord 招待リンクがあるか
- コピーライト表示があるか

#### 7. デザインの個性（AIっぽくないか）
- **スクリーンショットを見て**: 「AIが自動生成したサイト」に見えないか
- SPEC.md の「デザインの方向性」に書かれたコンセプト・リファレンスに沿っているか
- 禁止パターンに該当していないか:
  - 均等な角丸カード + 影の羅列
  - 全要素の中央揃え
  - 均一な余白の繰り返し
  - 汎用的なSaaSテンプレート感
- そのアプリならではの視覚的な特徴が1つ以上あるか
- 「このアプリを10個並べたら全部違って見えるか？」を基準にする

#### 8. 文言チェック
- ヘッドラインが簡潔で伝わるか（冗長・ポエムすぎないか）
- CTAの文言が行動を促すか（「送信」ではなく「診断する」のように具体的か）
- トーンが統一されているか（丁寧すぎ/カジュアルすぎのブレがないか）
- 不要な説明文がないか（ユーザーが読まないテキスト）
- メタデータ（title, description）がアプリの内容を正確に表しているか

各指摘は具体的なファイル名・行番号付き。

### ラウンド2: 修正

ラウンド1の指摘を修正:

- **レイアウト修正**: 余白・配置の調整
- **配色修正**: `tone.config.ts` を更新
- **文言修正**: ヘッドライン・CTA・説明文の書き直し
- **フッター修正**: `components/footer.tsx` を更新
- **ローディング/エラー**: 必要なコンポーネントを追加

修正後、`pnpm build` が通ることを確認。

### ラウンド3: 再キャプチャ・再チェック・スコアリング

**修正後のスクリーンショットを再度撮影**して、修正が反映されていることを目視確認。

```bash
npx playwright screenshot --browser chromium --full-page http://localhost:3000 /tmp/review-desktop-final.png
npx playwright screenshot --browser chromium --full-page --viewport-size "375,812" http://localhost:3000 /tmp/review-mobile-final.png
```

スクリーンショットを Read ツールで確認してからスコアリング:

| 指標 | 0 | 1 | 2 | 3 |
|------|---|---|---|---|
| **第一印象** | 何のアプリか不明 | 10秒で理解 | 5秒で理解 | 3秒で理解できる |
| **主要アクション到達** | 5クリック以上 | 4クリック | 3クリック | 1-2クリック |
| **配色・トーン** | テキストが読めない | 一部読みにくい | おおむね読みやすい | 完璧に読みやすい |
| **レスポンシブ** | モバイルで崩壊 | 一部崩れる | 使えるレベル | 完璧に対応 |
| **エラー・ローディング** | 表示なし | 一部対応 | 主要フローは対応 | 全フロー対応 |
| **フッター** | 未設置 | 一部欠落 | 全要素あり | 全要素あり+自然な導線 |
| **デザインの個性** | AIテンプレート感 | やや汎用的 | 個性はあるが弱い | そのアプリならではの特徴 |
| **文言** | 意味不明・ポエム | やや冗長 | 簡潔で伝わる | 行動を促す文言 |

## ゲート

**合計16点以上（24点満点）かつ 0点の項目がないこと**

不合格の場合:
- 0点の項目を優先修正
- 修正後にスクリーンショット再撮影 → 再スコアリング

## dev サーバー停止

レビュー完了後、dev サーバーを停止:

```bash
kill $(lsof -ti:3000) 2>/dev/null
```
