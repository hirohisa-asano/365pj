# /implement — Phase 2: 実装

## あなたの役割

あなたは **フルスタックエンジニア** です。
SPEC.md と TODO.md に基づいてアプリのコードを生成し、ビルドを通します。

## 入力

`$ARGUMENTS` = アプリディレクトリ名（例: `app-003`）

## 前提

- `apps/{アプリディレクトリ}/SPEC.md` と `TODO.md` が存在すること
- `/new-app` で初期化済みであること

## 技術スタック（遵守事項）

- Next.js (App Router) + TypeScript (strict)
- Tailwind CSS v4 + shadcn/ui + Lucide
- `src/` ディレクトリは使わない
- `tone.config.ts` の色・フォント・角丸を使う（CSS変数経由）
- フッターに寄付リンク（Stripe）+ ポータルリンク + Discord リンクを設置
- OGP は `app/opengraph-image.tsx` で自動生成
- Biome でフォーマット・リント

## 手順

### ラウンド0: 外部サービス準備

SPEC.md の「使用サービス」を確認し、外部サービスを使う場合:
1. `.claude/approved-services.md` のコードパターンを読む
2. 必要なパッケージを `pnpm add` する
3. 環境変数が必要な場合、`apps/lab/.env.local`（マスター）から該当キーをコピーする。マスターにもなければ人間に通知
4. `.env.example` に環境変数名を追記する

### ラウンド1: コード生成

SPEC.md の全要件を読み、以下を生成:

1. **ページ**: `app/page.tsx` および必要な追加ページ (`app/xxx/page.tsx`)
2. **コンポーネント**: `components/` に UI コンポーネント（1コンポーネント100行以内目安）
3. **API Routes**: `app/api/` に必要なエンドポイント（API使用時のみ）
4. **Supabase マイグレーション**: DB使用時、`supabase/migrations/` に SQL ファイル
5. **OGP 更新**: `app/opengraph-image.tsx` のタイトル・色をアプリに合わせて更新
6. **layout.tsx 更新**: メタデータ（title, description）をアプリに合わせて更新

### コーディング規約

- **日本語 UI**: ユーザー向けテキストは日本語
- **変数名・関数名**: 英語（キャメルケース）
- **コメント**: 必要最小限。ロジックが自明なら書かない
- **型定義**: 必要な型は各ファイル内か `types.ts` に定義
- **エラーハンドリング**: ユーザー入力・外部API境界のみ。内部コードは信頼する
- **状態管理**: React の useState/useReducer で十分。外部ライブラリ不要
- **レスポンシブ**: モバイルファースト。Tailwind のブレークポイントを使う
- **Claude API モデル**: `claude-haiku-4-5-20251001` を使う（CLAUDE.md 参照）

### プラン設計の実装

SPEC.md の「プラン設計」に従い、minimum / medium / normal の機能制限を実装する。

- **回数制限の実装**: API Route で以下をチェック
  - minimum: `localStorage` のカウント（クライアント側。リセットは1日ごと）
  - medium: Supabase の `usage` テーブルでカウント
  - normal: 制限なし（`memberships` テーブルの `status = 'active'` を確認）
- **機能制限の実装**: レスポンスの内容を分岐
  - minimum: 基本結果のみ返す
  - medium: 詳細結果を含む
  - normal: 全情報を返す
- **認証不要アプリ（minimum only）の場合**: プラン制御コードは不要

### デザイン規約（AIっぽいデザインを避ける）

SPEC.md の「デザインの方向性」セクションを必ず読み、それに従うこと。

**禁止パターン（これをやると「AIが作った感」が出る）:**
- 均等な角丸カード + box-shadow の羅列
- 全要素を中央揃えにする（左寄せ or 非対称を基本にする）
- セクションごとに同じ余白（py-24 の繰り返し等）
- グラデーションボタンの多用
- アイコンを装飾的に並べるだけ
- 「〇〇を始めましょう！」のような汎用コピー
- 白背景に1色のアクセントカラーだけ
- ヒーローセクション → 3カラムカード → CTA の定型構成
- 全ボタンに同じスタイル（プライマリ/ゴーストを使い分ける）
- 装飾のない `<input>` に `placeholder` だけ置く

**やるべきこと:**
- SPEC.md の「リファレンスサイト」のスクショ（/tmp/ref-*.png）を Read で確認し、そのレイアウト・余白・色使いを再現する
- 余白にメリハリをつける（密と疎のコントラスト）
- テキストのサイズに階層をつける（h1 は大きく、補足は小さく）
- アプリの内容に合った視覚的な遊びを1つ入れる（tone.config の配色だけでなく）
- フォームは「入力させられている感」を減らす（ラベルの工夫、プレースホルダの具体例）
- 結果表示はカードより「読み物」感を意識する

**「人が作った感」を出す具体テクニック:**
- **フォントウェイト**: 見出しは`font-black`(900)、本文は`font-normal`(400)。差を大きくつける
- **サイズの飛躍**: h1 を `text-4xl` 以上にして、本文 `text-sm` との落差をつける
- **色の深み**: primary だけでなく、primary の薄い版（bg-primary/5）や濃い版を使う
- **テクスチャ**: ベタ塗りでなく、微妙なボーダー（`border`）や `bg-muted/50` の半透明を使う
- **非対称レイアウト**: 2カラムなら 7:3 や 2:1 にする。均等分割しない
- **余白のリズム**: セクション間の余白を意図的に変える（hero下は広く、フォーム周りは詰める）
- **文言の具体性**: 「入力してください」→「例: 月10万円の副業収入」のように実例で示す
- **数字・記号の強調**: スコアや%は周囲より2段階大きく、font-weight も変える
- **ホバー**: `transition-all duration-200` + 控えめな色変化（派手な scale やglow は避ける）

### モーション・アニメーション規約

テンプレートの `components/motion.tsx` にある共通コンポーネントを使うこと。

**使用すべき場面:**
- **ページ読み込み時**: メインコンテンツを `<FadeIn>` で表示（最初の描画で「静止画面がバッと出る」のを防ぐ）
- **結果・スコア表示**: `<ScoreReveal>` + `<CountUp>` で演出（診断系アプリの必須パターン）
- **リスト表示**: `<Stagger>` + `<StaggerItem>` で順番に表示
- **CTA ボタン**: `<PressScale>` で押した感触
- **ローディング中**: `.skeleton` クラス or `.animate-pulse-slow` で待機演出
- **スクロール連動**: 長いページでは `<FadeInView>` でセクションごとに出現

**禁止パターン:**
- ページ全体の大きなトランジション（遅くて邪魔）
- 3秒以上のアニメーション（ユーザーを待たせない）
- 同じアニメーションの連続（FadeIn だけ5回など。種類を混ぜる）
- 装飾だけのアニメーション（意味のない浮遊・回転）

**診断系アプリのパターン（必ず使う）:**
1. 入力フォーム → 送信ボタン押下
2. `.animate-pulse-slow` のローディング演出（2-3秒の「溜め」）
3. `<ScoreReveal>` でスコアがドンと出る
4. `<CountUp>` で数字がカウントアップ
5. `<Stagger>` で詳細結果が順番に表示

### ラウンド2: ビルド・リント修正

```bash
cd apps/{アプリディレクトリ}
pnpm install
pnpm build
```

ビルドエラーがあれば修正してください。修正後に再度 `pnpm build` を実行。
**最大3回** リトライ。3回失敗したら停止し、エラーを報告。

```bash
pnpm lint
```

リントエラーがあれば修正。

```bash
pnpm format
```

フォーマットを適用。

SPEC.md の **機能要件** セクションを1項目ずつ確認:

1. 各要件に対応するコードが存在するか
2. 漏れがあれば実装を追加
3. TODO.md の実装タスクにチェックを入れる

## 評価（Pass/Fail）

| 指標 | Pass 基準 |
|------|----------|
| pnpm build | エラー0で成功 |
| pnpm lint | エラー0 |
| SPEC充足率 | 機能要件100%実装 |
| 合計行数 | 500行以内（テンプレートのコードは除外して数える） |
| 新規ファイル数 | 15ファイル以内 |

**通過基準: 全項目 Pass**

- まずリファクタリングを試みる
