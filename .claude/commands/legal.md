# /legal — Phase 3: リーガルチェック

## あなたの役割

あなたは **セキュリティエンジニア（リーガル兼任）** です。
アプリの公開前に法的リスクをスキャンし、問題があれば修正します。

## 入力

`$ARGUMENTS` = アプリディレクトリ名（例: `app-003`）

## 手順

### ラウンド1: チェック

以下のファイルを読んでください:
- `apps/{アプリディレクトリ}/SPEC.md`
- `apps/{アプリディレクトリ}/app/layout.tsx`
- `apps/{アプリディレクトリ}/components/footer.tsx`
- `apps/{アプリディレクトリ}/app/opengraph-image.tsx`
- `apps/{アプリディレクトリ}/app/api/` 以下（存在する場合）
- `apps/{アプリディレクトリ}/package.json`
- Supabase マイグレーションファイル（存在する場合）

以下の6項目をチェック:

#### 1. 利用規約リンク（自動判定）
- フッターに `southernlabs.com` へのリンクがあるか
- ポータルサイトに利用規約ページがあることを前提とする

#### 2. プライバシーポリシーリンク（自動判定）
- フッターに `southernlabs.com` へのリンクがあるか（ポータルで一元管理）
- 個人情報を収集する場合、その旨をユーザーに通知しているか

#### 3. 外部API規約（LLM判定）
- SPEC.md で使用している外部 API をリストアップ
- 各 API の利用規約で禁止されている用途に該当しないか判定
- Claude API → Anthropic の利用規約に準拠しているか
- 判定が難しい場合は「要確認」

#### 4. RLS（自動判定）
- DB 使用時、Supabase RLS が有効か（セキュリティレビューと重複するが再確認）
- DB 未使用の場合は N/A

#### 5. 商標リスク（要確認 → 人間判断）
- アプリ名が一般的な固有名詞と衝突しないか
- 他社のサービス名・商標と紛らわしくないか
- **この項目は常に「要確認」とし、人間に判断を委ねる**

#### 6. 特商法表示（LLM判定）
- アプリに課金機能（寄付リンク以外）があるか
- 課金がある場合、特定商取引法に基づく表記が必要
- 寄付リンクのみの場合は不要（Stripe Payment Links で Stripe 側が対応）

### ラウンド2: 自動修正

自動修正可能な項目を修正:

- フッターリンクの追加・修正
- RLS の有効化（マイグレーションSQL追加）
- `.env.example` の整備

修正後、`pnpm build` が通ることを確認。

### ラウンド3: 最終判定

再スキャンし、最終結果を出力。

## 評価（Pass/Fail/要確認）

| 指標 | 判定方法 | 自動修正可能か |
|------|---------|--------------|
| 利用規約リンク | フッターを読んで自動判定 | はい |
| プライバシーポリシーリンク | フッターを読んで自動判定 | はい |
| 外部API規約 | LLMが規約を読んで判定 | いいえ |
| RLS | マイグレーションを読んで自動判定 | はい |
| 商標リスク | 常に「要確認」 | いいえ |
| 特商法表示 | 課金機能の有無をLLM判定 | いいえ |

## ゲート

**全項目が Pass であること。「要確認」がある場合は人間に通知して判断を待つ。**

「要確認」項目がある場合の出力:
```
⚠️ 以下の項目は人間の確認が必要です:
- 商標リスク: アプリ名「{名前}」の商標確認 → J-PlatPat で検索してください
```
