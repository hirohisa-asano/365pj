# /review-all — Phase 2.5: 全レビュー一括実行

## 概要

3つのレビュー（設計・UI/UX・セキュリティ）を順番に実行します。
すべてのレビューを通過したら、最終ビルド確認を行います。

## 入力

`$ARGUMENTS` = アプリディレクトリ名（例: `app-003`）

## 実行順序

```
① アーキテクトレビュー → ② UI/UXデザインレビュー → ③ セキュリティレビュー → 最終ビルド
```

各レビューが不合格の場合、修正してから次に進みます。

---

## ① アーキテクトレビュー（設計レビュー）

### 役割: アーキテクト

全ソースコードと SPEC.md を読み、以下5観点で評価:

**チェック対象**: `apps/$ARGUMENTS/` 以下の全 `.ts`, `.tsx` ファイル + SPEC.md

#### 観点と評価基準

| 指標 | 0 | 1 | 2 | 3 |
|------|---|---|---|---|
| SPEC整合性 | 大きな乖離 | 部分的に不一致 | 軽微な差異 | 完全一致 |
| コンポーネント分割 | 1ファイル300行超 | 一部200行超 | おおむね適切 | 各100行以内 |
| データフロー | props 5階層超 | 3-4階層 | 2階層以内 | 状態管理最小限 |
| SSG/SSR選択 | 明らかに不適切 | 一部不適切 | おおむね適切 | 最適 |
| 依存パッケージ | 不要3個以上 | 1-2個不要 | 全て必要 | 必要最小限 |

**ゲート: 合計10/15以上、0点項目なし**

#### 手順
1. 全ファイルを読み、指摘を洗い出す
2. 指摘を修正する
3. 再チェック・スコアリング → `REVIEW-architecture.md` に出力

---

## ② UI/UXデザインレビュー

### 役割: UI/UXデザイナー

tone.config.ts・レイアウト・ユーザビリティを評価:

**チェック対象**: tone.config.ts, globals.css, 全ページ・コンポーネント, footer.tsx

#### 観点と評価基準

| 指標 | 0 | 1 | 2 | 3 |
|------|---|---|---|---|
| 第一印象 | 何のアプリか不明 | 10秒で理解 | 5秒で理解 | 3秒で理解 |
| 主要アクション到達 | 5クリック以上 | 4クリック | 3クリック | 1-2クリック |
| 配色・トーン | 不一致 | 一部違和感 | おおむね合う | 完璧にマッチ |
| レスポンシブ | 崩壊 | 一部崩れ | 使えるレベル | 完璧 |
| エラー・ローディング | 表示なし | 一部対応 | 主要フロー対応 | 全フロー対応 |
| フッター | 未設置 | 一部欠落 | 全要素あり | 全要素+自然な導線 |

**ゲート: 合計12/18以上、0点項目なし**

#### 手順
1. 全ファイルを読み、指摘を洗い出す
2. tone.config.ts・Tailwindクラス・コンポーネントを修正
3. 再チェック・スコアリング → `REVIEW-design.md` に出力

---

## ③ セキュリティレビュー

### 役割: セキュリティエンジニア

脆弱性をスキャンし、CRITICAL は即修正:

**チェック対象**: 全ソースコード + package.json + .env.example

#### 観点（全て Pass/Fail）

| 指標 | Pass 基準 | 該当条件 |
|------|----------|---------|
| XSS | dangerouslySetInnerHTML なし、ユーザー入力サニタイズ済み | 常にチェック |
| API入力検証 | 全エンドポイントに型チェックあり | API Routes 使用時 |
| RLS | RLS有効+ポリシー定義済み | DB使用時 |
| 環境変数 | ハードコード0件 | 常にチェック |
| APIキー露出 | クライアント側に秘密キー0件 | 常にチェック |

**ゲート: 全該当項目 Pass**

#### 手順
1. 全ファイルをスキャン、脆弱性を洗い出す
2. CRITICAL → 即修正、WARNING → 可能な限り修正
3. 再スキャン → `REVIEW-security.md` に出力

---
