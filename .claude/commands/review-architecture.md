# /review-architecture — Phase 2.5: 設計レビュー

## あなたの役割

あなたは **アーキテクト** です。
実装されたコードの設計品質を評価し、問題があれば修正します。
あなたはコードを書いた人物ではありません。独立した第三者の視点でレビューしてください。

## 入力

`$ARGUMENTS` = アプリディレクトリ名（例: `app-003`）

## 手順

### ラウンド1: 指摘

以下のファイルをすべて読んでください:
- `apps/{アプリディレクトリ}/SPEC.md`
- `apps/{アプリディレクトリ}/app/` 以下の全ファイル
- `apps/{アプリディレクトリ}/components/` 以下の全ファイル
- `apps/{アプリディレクトリ}/lib/` 以下の全ファイル（存在する場合）
- `apps/{アプリディレクトリ}/package.json`

以下の5つの観点で問題を洗い出してください:

#### 1. SPEC整合性
- SPEC.md の機能要件と実装が一致しているか
- 画面構成が SPEC 通りか
- SPEC にない機能が追加されていないか（スコープクリープ）

#### 2. コンポーネント分割
- 1ファイルが200行を超えていないか
- 1コンポーネントが複数の責務を持っていないか
- 再利用可能な部分が適切に分離されているか

#### 3. データフロー
- props のバケツリレーが3階層を超えていないか
- 状態管理が最小限か（不要な useState がないか）
- Server Component / Client Component の境界が適切か

#### 4. SSG/SSR選択
- 静的で良いページが SSR になっていないか
- 動的データが必要なページが SSG になっていないか
- `"use client"` が必要な箇所にのみ付いているか

#### 5. 依存パッケージ
- `package.json` に不要なパッケージがないか
- 使っていない import がないか
- shadcn/ui で代替できるのに独自実装しているものがないか

各指摘は以下の形式で記録:
```
- [SPEC整合性] app/page.tsx:25 — SPEC にある「検索機能」が未実装
- [コンポーネント分割] components/main.tsx — 350行。分割すべき
```

### ラウンド2: 修正

ラウンド1で見つかった指摘を **上から順に修正** してください。
修正後、`pnpm build` が通ることを確認。

### ラウンド3: 再チェック・スコアリング

修正後のコードを再度同じ5観点でチェック。
残存する問題がないことを確認したら、スコアリング:

| 指標 | 0 | 1 | 2 | 3 |
|------|---|---|---|---|
| **SPEC整合性** | 大きな乖離がある | 部分的に不一致 | 軽微な差異 | 完全一致 |
| **コンポーネント分割** | 1ファイル300行超 | 一部200行超 | おおむね適切 | 各100行以内で明確 |
| **データフロー** | props 5階層超 | 3-4階層 | 2階層以内 | 状態管理が最小限 |
| **SSG/SSR選択** | 明らかに不適切 | 一部不適切 | おおむね適切 | 最適 |
| **依存パッケージ** | 不要なもの3個以上 | 1-2個不要 | 全て必要 | 必要最小限 |

## ゲート

**合計10点以上（15点満点）かつ 0点の項目がないこと**

不合格の場合:
- 0点の項目を優先的に修正
- 修正後に再スコアリング
- 2回修正しても通過しない場合、その旨を報告
