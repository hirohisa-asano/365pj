# 全自動パイプライン計画

365PJの「需要分析→アイデア選定→公開→マーケティング」を自動化するパイプライン。

## 全体像

```
需要分析 → 計画立案 → 設計・実装 → リーガル → 公開 → マーケティング
  (0)       (1)         (2)         (3)      (4)        (5)
```

最終的に毎日実行できる仕組みにする。

## 実行方法

Claude Code の非対話モードで、途中で止まらずに実行する:

```bash
claude -p "PIPELINE.md に基づいて今日のアプリを作ってデプロイして" \
  --max-turns 200
```

| オプション | 効果 |
|-----------|------|
| `--max-turns N` | 最大Nターンまで自動継続（途中で止まらない） |
| `--allowedTools` | 許可するツールを指定（確認プロンプトをスキップ） |
| `-p "prompt"` | 非対話モード（パイプラインに組み込める） |

## フェーズ0: 需要分析（自動アイデア候補生成）

無料データソースから需要を自動収集し、アイデア候補を生成する。

### データソース（すべて無料）

| ソース | 取れるデータ | 用途 |
|--------|-------------|------|
| Google Trends | 急上昇キーワード | 伸びてる領域の検知 |
| はてなブックマーク | テック系ホットエントリ + コメント | 日本エンジニアの不満・課題抽出 |
| Product Hunt | 直近ローンチで伸びてるツール | 海外で流行ってる→日本未上陸の発見 |
| GitHub Trending | 今週伸びてるリポジトリ | 技術トレンドの先行指標 |
| Zenn トレンド | 日本エンジニアの関心記事 | 関心事・困りごとの把握 |

### パイプライン

```
1. 各データソースから直近データを取得
2. LLM で集約分析
   - 「繰り返し出てくる不満・課題」を抽出
   - ideas-detail.md のフォーマットでアイデア候補を生成
   - 既存アイデア（ideas-detail.md）との重複チェック
   - 競合の強さをスコアリング
3. ideas-candidates.md に候補を追記
4. 人間が見て「作る/作らない」を判断
```

### 将来拡張（有料）

- X API Basic（$100/月）: 「〇〇 欲しい」「〇〇 使いにくい」で日本語ツイート検索
- 無料ソースで価値を確認してから追加する

## フェーズ1: 計画立案（アイデア → 設計書の自動生成）

アイデア番号を指定したら、設計書・タスク分解・tone.config まで自動生成。

```
入力: アイデア番号 (#27 LLMコスト計算)
↓
出力:
  - apps/app-NNN/ をテンプレートからコピー
  - SPEC.md（ユーザー・課題・画面構成・API設計）
  - tone.config.ts（アプリの雰囲気に合った配色自動選定）
  - TODO.md（タスク分解、3日以内に収まる粒度）
```

`ideas-detail.md` の「誰が・課題・理想・競合状況」をそのまま設計のインプットにする。

---

## フェーズ2: 実装（設計書 → コード生成）

SPEC.md に基づいてコードを自動生成。

- page.tsx, コンポーネント, API Routes を SPEC.md から生成
- Supabase マイグレーション SQL の自動生成
- `pnpm build` で壊れたら自動修正ループ

現実的な落とし所: **80%自動生成 → 人間が確認・調整**。

---

## フェーズ3: リーガルチェック

公開前にリーガルリスクを自動スキャン。

```
チェック項目:
  □ 利用規約・プライバシーポリシーへのリンクがフッターにあるか
  □ 外部APIの利用規約に違反していないか
  □ 個人情報を収集する場合、Supabase RLS が有効か
  □ 商標リスク（アプリ名の商標検索）
  □ OGP画像に他者の著作物を含んでいないか
  □ 特定商取引法の表示が必要か（課金がある場合）
```

判定が微妙なものだけ人間に確認を求める。

## フェーズ4: 公開（デプロイ自動化）

コマンド一発で Vercel + Supabase + DNS まで完了。

```bash
# deploy.sh app-003 burnrate
1. Vercel プロジェクト作成 (vercel CLI)
2. 環境変数セット (vercel env)
3. Supabase スキーマ作成 (supabase migration)
4. サブドメイン設定 (vercel domains add burnrate.southernlabs.com)
5. デプロイ (vercel --prod)
6. ポータル (apps/lab) にリンク追加
7. publish-log.md に公開記録追加
```

---

## フェーズ5: マーケティング・PR

### Level 1: テキスト自動生成

```
入力: SPEC.md + デプロイURL
↓
出力:
  - X (Twitter) 投稿文（140字以内、ハッシュタグ付き）
  - Zenn / note 記事のドラフト（開発背景・技術スタック）
  - Product Hunt 投稿用テキスト
```

### Level 2: 半自動投稿（API連携）

- X API で下書き作成 → 人間が確認して投稿
- Zenn CLI でドラフト記事を自動生成

### Level 3: 反応トラッキング

- Vercel Analytics のデータを定期取得
- `publish-log.md` に PV・直帰率を自動追記
- 寄付数 + PV で「育てるべきアプリ」をスコアリング

## 実装順序

| 優先度 | フェーズ | 理由 | 状態 |
|--------|----------|------|------|
| **1** | フェーズ4: 公開自動化 | 毎回やる手作業が最も多い。即効性が高い | 未着手 |
| **2** | フェーズ0: 需要分析 | 無料ソースで始められる。アイデアの質が上がる | 未着手 |
| **3** | フェーズ1: 計画自動化 | アイデア→設計の変換。Claude Code スキルで作れる | 未着手 |
| **4** | フェーズ5-L1: PR文生成 | テキスト生成だけなら即日実装可能 | 未着手 |
| **5** | フェーズ3: リーガル | チェックリスト自動化 | 未着手 |
| **6** | フェーズ2: 実装自動化 | 最も難易度が高い。品質担保が課題 | 未着手 |
| **7** | 毎日実行パイプライン | 全スクリプトを cron/scheduled agent で結合 | 未着手 |

## スクリプト設計方針

毎日自動実行に向けて、各スクリプトは以下を守る:

1. **対話的な入力を求めない**（CLI引数 or 設定ファイルで完結）
2. **冪等にする**（2回実行しても壊れない）

```
365pj/
├── scripts/
│   ├── demand-scan.ts     ← フェーズ0: 需要分析（無料ソース収集+LLM分析）
│   ├── init-app.ts        ← フェーズ1: アイデア→アプリ初期化
│   ├── deploy.sh          ← フェーズ4: ワンコマンドデプロイ
│   ├── legal-check.ts     ← フェーズ3: リーガルチェック
│   ├── generate-pr.ts     ← フェーズ5: マーケ文生成
│   └── analytics.ts       ← フェーズ5: 反応トラッキング
├── .claude/
│   └── commands/
│       ├── new-app.md     ← /new-app でアプリ初期化
│       ├── ship.md        ← /ship でデプロイ+PR生成
│       └── legal.md       ← /legal でリーガルチェック
```

## マネタイズ

- Stripe Payment Links は**共通リンク1本**を維持
- `client_reference_id` パラメータでアプリを識別する
  - 例: `https://buy.stripe.com/xxxxx?client_reference_id=app-003`
- フッターのリンク生成時に自動でパラメータを付与
- Stripe ダッシュボードでアプリ別の寄付額を確認可能

## フィードバック・バグ報告（Discord）

既存の Discord サーバーを活用する。

### チャンネル構成

```
既存Discordサーバー
├── #お知らせ          ← 新アプリ公開時に Webhook で自動通知
├── #バグ報告          ← ユーザーからのバグ報告（アプリ名タグ付き）
├── #リクエスト        ← 「こんなの欲しい」→ 需要分析のインプットにもなる
└── #開発ログ          ← デプロイ時に自動で進捗投稿
```

### 自動化

- デプロイ完了時 → `#お知らせ` に Webhook で自動投稿（アプリ名・URL・概要）
- `#バグ報告` の投稿 → GitHub Issues に Bot で自動起票
- `#リクエスト` の投稿 → フェーズ0（需要分析）のインプットとして活用
- 各アプリのフッターに Discord 招待リンクを設置

---

## 「全自動」と「半自動」の線引き

- **全自動**: ファイルコピー、環境変数設定、デプロイ、OGP生成、ログ記録
- **半自動（生成→確認→実行）**: SNS投稿、リーガル判断、アプリ名決定、設計の最終判断
- **手動のまま**: アイデア選定の最終判断、課金モデルの決定

> 「考える時間」は人間に残して「作業時間」を0にする。
