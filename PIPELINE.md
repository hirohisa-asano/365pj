# 全自動パイプライン計画

365PJの「需要分析→公開→マーケティング」を自動化するパイプライン。

## 基本方針

- **半自動**: 候補を自動生成し、人間が選ぶ。ただし手動でアイデア指定も可能
- **リリース**: 人間の確認不要で自動デプロイ。ただし PR（マーケ投稿）前には人間が確認
- **実行基盤**: すべて Claude Code（`--max-turns`）で回す。API スクリプトには切り出さない
- **品質担保**: 役割別レビューで段階的に品質を上げる
- **各フェーズ3ラウンド**: 生成→評価→改善の3ターンで精度を上げる

## 全体像

```
需要分析 → 計画立案 → 実装 → レビュー → リーガル → 公開 → マーケティング
  (0)       (1)      (2)    (2.5)     (3)      (4)       (5)
```

## 実行方法

```bash
claude -p "PIPELINE.md に基づいて今日のアプリを作ってデプロイして" \
  --max-turns 200
```

| オプション | 効果 |
|-----------|------|
| `--max-turns N` | 最大Nターンまで自動継続（途中で止まらない） |
| `--allowedTools` | 許可するツールを指定（確認プロンプトをスキップ） |
| `-p "prompt"` | 非対話モード（パイプラインに組み込める） |

---

## 役割定義

パイプライン全体で **7つの役割** を Claude Code が切り替えて担当する。

| # | 役割 | 担当フェーズ | 責務 |
|---|------|-------------|------|
| 1 | **マーケットリサーチャー** | 0: 需要分析 | データ収集・トレンド分析・需要スコアリング |
| 2 | **プロダクトマネージャー** | 1: 計画立案 | SPEC.md 作成・スコープ管理・実現可能性判定 |
| 3 | **フルスタックエンジニア** | 2: 実装 | コード生成・ビルド・デバッグ |
| 4 | **アーキテクト** | 2.5: レビュー | 設計の妥当性・コンポーネント構成・データフロー |
| 5 | **UI/UXデザイナー** | 2.5: レビュー | ビジュアル・ユーザビリティ・アクセシビリティ |
| 6 | **セキュリティエンジニア** | 2.5+3: レビュー+リーガル | 脆弱性・RLS・API露出・法的チェック |
| 7 | **マーケター** | 5: マーケティング | PR文生成・SNS投稿・反応分析 |

## フェーズ0: 需要分析（マーケットリサーチャー）

### データソース（すべて無料）

| ソース | 取れるデータ | 用途 |
|--------|-------------|------|
| Google Trends | 急上昇キーワード | 伸びてる領域の検知 |
| はてなブックマーク | テック系ホットエントリ + コメント | 日本エンジニアの不満・課題抽出 |
| Product Hunt | 直近ローンチで伸びてるツール | 海外で流行ってる→日本未上陸の発見 |
| GitHub Trending | 今週伸びてるリポジトリ | 技術トレンドの先行指標 |
| Zenn トレンド | 日本エンジニアの関心記事 | 関心事・困りごとの把握 |

### 3ラウンド

| ラウンド | やること | 出力 |
|----------|---------|------|
| R1: 収集 | 各データソースから直近データを取得。生データを整理 | `raw-trends.md`（生データ） |
| R2: 分析 | 繰り返し出てくる不満・課題を抽出。アイデア候補を生成。既存アイデアとの重複チェック | `ideas-candidates.md`（候補リスト+スコア） |
| R3: 精査 | 候補をスコアリングし直し。上位3つに絞り込み。人間に提示 | 上位3候補（手動指定も可） |

### 評価スコア（各項目 1〜5、合計25点満点）

| 指標 | 1（低い） | 5（高い） |
|------|----------|----------|
| **需要の強さ** | 言及がほぼない | 複数ソースで繰り返し出現 |
| **競合の弱さ** | 大手が参入済み | 専用ツールが存在しない |
| **3日で作れるか** | 外部API契約必須・複雑なDB設計 | 公開API+シンプルUI |
| **バイラル性** | 説明が必要 | 結果をシェアしたくなる |
| **課金ポテンシャル** | 無料で十分 | 有料版への導線が自然 |

**通過基準: 合計15点以上**

## フェーズ1: 計画立案（プロダクトマネージャー）

### 3ラウンド

| ラウンド | やること | 出力 |
|----------|---------|------|
| R1: 設計 | ideas-detail.md のインプットから SPEC.md ドラフトを生成。画面構成・API設計・使用技術を決定 | `SPEC.md` v1 |
| R2: 検証 | SPEC.md をスコアリング。スコープ超過・API制約違反がないかチェック。問題があれば修正 | `SPEC.md` v2（修正済み） |
| R3: 確定 | タスク分解・tone.config.ts 生成。テンプレートコピー。最終スコア算出 | `SPEC.md` 確定版 + `TODO.md` + `tone.config.ts` |

### 入力 → 出力

```
入力: アイデア番号 (#27 LLMコスト計算) or 手動指定テキスト
↓
出力:
  - apps/app-NNN/ をテンプレートからコピー
  - SPEC.md（ユーザー・課題・画面構成・API設計・評価スコア）
  - tone.config.ts（アプリの雰囲気に合った配色自動選定）
  - TODO.md（タスク分解、3日以内に収まる粒度）
```

### 評価スコア（各項目 1〜5、合計30点満点）

| 指標 | 1（低い） | 5（高い） |
|------|----------|----------|
| **課題の明確さ** | 誰の何の課題か曖昧 | ペルソナ・課題・理想が具体的 |
| **スコープの適切さ** | 3日では無理 / 小さすぎて価値なし | 3日で作れて価値がある |
| **画面設計の具体性** | 「いい感じに」レベル | 画面数・要素・遷移が明確 |
| **技術的実現性** | 未知の技術・API契約必要 | 既存スタックで完結 |
| **差別化** | ChatGPTで十分 | 専用UXで明確に勝てる |
| **収益化導線** | 寄付しか手段なし | 有料版・B2B展開が見える |

**通過基準: 合計20点以上**

## フェーズ2: 実装（フルスタックエンジニア）

### 3ラウンド

| ラウンド | やること | 出力 |
|----------|---------|------|
| R1: 生成 | SPEC.md + TODO.md に基づきコード生成。page.tsx・コンポーネント・API Routes・Supabase マイグレーション | 動くコード一式 |
| R2: ビルド修正 | `pnpm build` を実行。エラーがあれば修正。lint (`pnpm lint`) も通す | ビルド成功 |
| R3: セルフチェック | SPEC.md の全要件を1つずつ確認。漏れがあれば実装。TODO.md を全チェック | 要件充足率100% |

### 評価スコア（各項目 Pass/Fail）

| 指標 | Pass 基準 |
|------|----------|
| **ビルド** | `pnpm build` がエラー0で通る |
| **Lint** | `pnpm lint` がエラー0で通る |
| **SPEC充足率** | SPEC.md の機能要件を100%実装 |
| **行数** | 合計500行以内（小規模ルール） |
| **ファイル数** | 新規作成15ファイル以内 |

**通過基準: 全項目 Pass**

## フェーズ2.5: 役割別レビュー（品質ゲート）

実装後、デプロイ前に **3つの役割** が順番にレビューする。
各レビューは **3ラウンド**（指摘→修正→再チェック）で回す。

### レビューの流れ

```
実装完了（フェーズ2 Pass）
  ↓
① アーキテクトレビュー（3ラウンド）
  ↓
② UI/UXデザインレビュー（3ラウンド）
  ↓
③ セキュリティレビュー（3ラウンド）
  ↓
全レビュー通過 + pnpm build Pass → デプロイへ
```

### ① アーキテクトレビュー

| ラウンド | やること |
|----------|---------|
| R1: 指摘 | 全ファイルを読み、設計上の問題を洗い出す |
| R2: 修正 | 指摘を自動修正する |
| R3: 再チェック | 修正後に同じ観点で再チェック。残存問題がなければ Pass |

**評価スコア（各項目 0〜3点、合計15点満点）**

| 指標 | 0 | 1 | 2 | 3 |
|------|---|---|---|---|
| **SPEC整合性** | 大きな乖離 | 部分的に不一致 | 軽微な差異 | 完全一致 |
| **コンポーネント分割** | 1ファイル500行超 | 一部肥大化 | おおむね適切 | 各100行以内で明確 |
| **データフロー** | props 5階層超 | 3-4階層 | 2階層以内 | 状態管理が最小限 |
| **SSG/SSR選択** | 不適切 | 一部不適切 | おおむね適切 | 最適 |
| **依存パッケージ** | 不要なもの3個以上 | 1-2個不要 | 全て必要 | 必要最小限 |

**通過基準: 合計10点以上 かつ 0点の項目がない**

出力: `REVIEW-architecture.md`（スコア + 指摘 + 修正履歴）

### ② UI/UXデザインレビュー

| ラウンド | やること |
|----------|---------|
| R1: 指摘 | tone.config・レイアウト・ユーザビリティを評価 |
| R2: 修正 | 指摘を自動修正する |
| R3: 再チェック | 修正後に再チェック。残存問題がなければ Pass |

**評価スコア（各項目 0〜3点、合計18点満点）**

| 指標 | 0 | 1 | 2 | 3 |
|------|---|---|---|---|
| **第一印象** | 何のアプリか不明 | 10秒で理解 | 5秒で理解 | 3秒で理解 |
| **主要アクション到達** | 5クリック以上 | 4クリック | 3クリック | 1-2クリック |
| **配色・トーン** | アプリの雰囲気と不一致 | 一部違和感 | おおむね合っている | 完璧にマッチ |
| **レスポンシブ** | モバイルで崩壊 | 一部崩れる | 使えるレベル | 完璧に対応 |
| **エラー・ローディング** | 表示なし | 一部対応 | 主要フローは対応 | 全フロー対応 |
| **フッター** | 未設置 | 一部欠落 | 全要素あり | 全要素あり+自然な導線 |

**通過基準: 合計12点以上 かつ 0点の項目がない**

出力: `REVIEW-design.md`（スコア + 指摘 + 修正履歴）

### ③ セキュリティレビュー

| ラウンド | やること |
|----------|---------|
| R1: 指摘 | 脆弱性・情報露出をスキャン |
| R2: 修正 | 指摘を自動修正する |
| R3: 再チェック | 修正後に再チェック。残存問題がなければ Pass |

**評価スコア（各項目 Pass/Fail — セキュリティは妥協しない）**

| 指標 | Pass 基準 |
|------|----------|
| **XSS** | ユーザー入力を表示する箇所が全てサニタイズされている |
| **API入力検証** | API Route に入力バリデーションがある（使用時のみ） |
| **RLS** | Supabase テーブルに RLS が有効（DB使用時のみ） |
| **環境変数** | `.env.local` に閉じている。コードにハードコードなし |
| **APIキー露出** | `NEXT_PUBLIC_` に秘密キーが含まれていない |

**通過基準: 全項目 Pass**

出力: `REVIEW-security.md`（チェック結果 + 修正履歴）

**通過基準: 全項目 Pass。「要確認」がある場合は人間に通知して判断を待つ**

## フェーズ4: 公開（フルスタックエンジニア）

全レビュー通過後、自動でデプロイする。人間の確認不要。

```bash
# 自動実行される手順:
1. Vercel プロジェクト作成 (vercel CLI)
2. 環境変数セット (vercel env)
3. Supabase スキーマ作成 (supabase migration)（DB使用時のみ）
4. サブドメイン設定 (vercel domains add {name}.southerncrosslab.com)
5. デプロイ (vercel --prod)
6. ポータル (apps/lab) にリンク追加
7. publish-log.md に公開記録追加
8. Discord #お知らせ に Webhook で通知
```

### 評価スコア（各項目 Pass/Fail）

| 指標 | Pass 基準 |
|------|----------|
| **デプロイ成功** | Vercel デプロイが成功（200 レスポンス） |
| **本番URL疎通** | `{name}.southerncrosslab.com` にアクセスできる |
| **OGP** | `opengraph-image.tsx` が正常に画像を返す |
| **ポータル更新** | `apps/lab` にリンクが追加されている |
| **Discord通知** | `#お知らせ` に投稿されている |

**通過基準: 全項目 Pass**

| ラウンド | やること | 出力 |

### 出力物

```
MARKETING.md:
  - X (Twitter) 投稿文（140字以内、ハッシュタグ付き）
  - Zenn / note 記事ドラフト（開発背景・技術スタック）
```

### 評価スコア（各媒体ごとに 各項目 1〜5、合計20点満点）

| 指標 | 1（低い） | 5（高い） |
|------|----------|----------|
| **フック** | 何の話か分からない | 最初の1行で興味を引く |
| **課題共感** | 課題が伝わらない | 「あるある」と思える |
| **価値提示** | 機能列挙 | 使う前→使った後が想像できる |
| **CTA** | リンクなし | 「試してみて」+URL+明確な行動 |

**通過基準: 合計14点以上**

**注意: PR文は人間が確認してから投稿する（全自動にしない）**

## コマンド設計

全て Claude Code カスタムコマンド（`.claude/commands/`）で実装。
`scripts/` は使わない — Claude Code で完結する。

設計原則:
1. **対話的な入力を求めない**（`$ARGUMENTS` で完結）
2. **冪等にする**（2回実行しても壊れない）
3. **スコアを出力する**（SCORECARD.md に記録）

```
365pj/
├── templates/
│   └── SCORECARD.md                ← スコアカードテンプレート
├── .claude/
│   └── commands/
│       ├── pipeline.md             ← /pipeline 全フェーズ一括実行
│       ├── demand-scan.md          ← /demand-scan 需要分析
│       ├── new-app.md              ← /new-app アプリ初期化
│       ├── implement.md            ← /implement 実装
│       ├── review-architecture.md  ← /review-architecture 設計レビュー
│       ├── review-design.md        ← /review-design UI/UXレビュー
│       ├── review-security.md      ← /review-security セキュリティレビュー
│       ├── review-all.md           ← /review-all 全レビュー一括
│       ├── legal.md                ← /legal リーガルチェック
│       ├── ship.md                 ← /ship デプロイ
│       └── marketing.md            ← /marketing PR文生成
```

### コマンド一覧

| コマンド | フェーズ | 入力例 |
|----------|---------|--------|
| `/demand-scan` | 0: 需要分析 | `/demand-scan` or `/demand-scan AI` |
| `/new-app` | 1: 計画立案 | `/new-app #27` or `/new-app LLMコスト計算ツール` |
| `/implement` | 2: 実装 | `/implement app-003` |
| `/review-architecture` | 2.5: 設計レビュー | `/review-architecture app-003` |

## マネタイズ

- Stripe Payment Links は**共通リンク1本**を維持
- `client_reference_id` パラメータでアプリを識別する
  - 例: `https://buy.stripe.com/xxxxx?client_reference_id=app-003`
- フッターのリンク生成時に自動でパラメータを付与
- Stripe ダッシュボードでアプリ別の寄付額を確認可能

## フィードバック・バグ報告（Discord）

既存の Discord サーバーを活用する。

### チャンネル構成

```
既存Discordサーバー
├── #お知らせ          ← 新アプリ公開時に Webhook で自動通知
├── #バグ報告          ← ユーザーからのバグ報告（アプリ名タグ付き）
├── #リクエスト        ← 「こんなの欲しい」→ 需要分析のインプットにもなる
└── #開発ログ          ← デプロイ時に自動で進捗投稿
```

### 自動化

- デプロイ完了時 → `#お知らせ` に Webhook で自動投稿（アプリ名・URL・概要）
- `#バグ報告` の投稿 → GitHub Issues に Bot で自動起票
- `#リクエスト` の投稿 → フェーズ0（需要分析）のインプットとして活用
- 各アプリのフッターに Discord 招待リンクを設置

## 外部 API 連携ポリシー

### パイプライン自体が使う API

| 用途 | API | キー管理 |
|------|-----|----------|
| 需要分析 | Google Trends, はてブ, Product Hunt, GitHub | スクレイピング or 公開API。キー不要 or 無料枠 |
| デプロイ | Vercel CLI, Supabase CLI | ローカルに認証済み |
| Discord通知 | Discord Webhook | `.env` に URL を1回設定 |
| マーケ投稿 | X API（将来） | 追加時に `.env` にキー追加 |

### 各アプリが使う API（3層分類）

| 層 | 例 | 対応 |
|----|-----|------|
| **共通（既に持ってる）** | Claude API, Supabase | 全アプリ共通の `.env` から自動セット |
| **無料・キー不要** | 公開RSS, スクレイピング, 公開API | そのまま使う |
| **新規契約が必要** | 特殊な有料API | **そのアプリでは使わない設計にする** |

### SPEC.md 生成時のルール

- 使える API: **Claude API, Supabase, 公開API/RSS のみ**
- 新規APIキーが必要な設計は避ける（例: Amazon API → Claude の知識で代替）
- どうしても必要な場合は「要API契約」フラグを立てて人間に判断を委ねる
- 3日ルールを守るため、API契約の手間が発生する設計を選ばない

## 「全自動」と「半自動」の線引き

- **全自動**: 実装、レビュー+自動修正、リーガルチェック、デプロイ、OGP生成、Discord通知、スコア記録
- **半自動（候補提示→人間が選ぶ）**: アイデア選定（上位3候補から選択 or 手動指定）
- **半自動（生成→確認→実行）**: SNS投稿・マーケ記事（PR前に人間が確認）
- **手動のまま**: 課金モデルの決定

> 「考える時間」は人間に残して「作業時間」を0にする。
