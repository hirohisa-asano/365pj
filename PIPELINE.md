# 全自動パイプライン計画

365PJの「アイデア選定→公開→マーケティング」を自動化するパイプライン。

## 全体像

```
アイデア選定 → 設計・実装 → リーガル → 公開 → マーケティング
   (1)          (2)         (3)      (4)        (5)
```

最終的に毎日実行できる仕組みにする。

---

## フェーズ1: 計画立案（アイデア → 設計書の自動生成）

アイデア番号を指定したら、設計書・タスク分解・tone.config まで自動生成。

```
入力: アイデア番号 (#27 LLMコスト計算)
↓
出力:
  - apps/app-NNN/ をテンプレートからコピー
  - SPEC.md（ユーザー・課題・画面構成・API設計）
  - tone.config.ts（アプリの雰囲気に合った配色自動選定）
  - TODO.md（タスク分解、3日以内に収まる粒度）
```

`ideas-detail.md` の「誰が・課題・理想・競合状況」をそのまま設計のインプットにする。

---

## フェーズ2: 実装（設計書 → コード生成）

SPEC.md に基づいてコードを自動生成。

- page.tsx, コンポーネント, API Routes を SPEC.md から生成
- Supabase マイグレーション SQL の自動生成
- `pnpm build` で壊れたら自動修正ループ

現実的な落とし所: **80%自動生成 → 人間が確認・調整**。

---

## フェーズ3: リーガルチェック

公開前にリーガルリスクを自動スキャン。

```
チェック項目:
  □ 利用規約・プライバシーポリシーへのリンクがフッターにあるか
  □ 外部APIの利用規約に違反していないか
  □ 個人情報を収集する場合、Supabase RLS が有効か
  □ 商標リスク（アプリ名の商標検索）
  □ OGP画像に他者の著作物を含んでいないか
  □ 特定商取引法の表示が必要か（課金がある場合）
```

判定が微妙なものだけ人間に確認を求める。

---

## フェーズ4: 公開（デプロイ自動化）

コマンド一発で Vercel + Supabase + DNS まで完了。

```bash
# deploy.sh app-003 burnrate
1. Vercel プロジェクト作成 (vercel CLI)
2. 環境変数セット (vercel env)
3. Supabase スキーマ作成 (supabase migration)
4. サブドメイン設定 (vercel domains add burnrate.southernlabs.com)
5. デプロイ (vercel --prod)
6. ポータル (apps/lab) にリンク追加
7. publish-log.md に公開記録追加
```

---

## フェーズ5: マーケティング・PR

### Level 1: テキスト自動生成

```
入力: SPEC.md + デプロイURL
↓
出力:
  - X (Twitter) 投稿文（140字以内、ハッシュタグ付き）
  - Zenn / note 記事のドラフト（開発背景・技術スタック）
  - Product Hunt 投稿用テキスト
```

### Level 2: 半自動投稿（API連携）

- X API で下書き作成 → 人間が確認して投稿
- Zenn CLI でドラフト記事を自動生成

### Level 3: 反応トラッキング

- Vercel Analytics のデータを定期取得
- `publish-log.md` に PV・直帰率を自動追記
- 寄付数 + PV で「育てるべきアプリ」をスコアリング

---

## 実装順序

| 優先度 | フェーズ | 理由 | 状態 |
|--------|----------|------|------|
| **1** | フェーズ4: 公開自動化 | 毎回やる手作業が最も多い。即効性が高い | 未着手 |
| **2** | フェーズ1: 計画自動化 | アイデア→設計の変換。Claude Code スキルで作れる | 未着手 |
| **3** | フェーズ5-L1: PR文生成 | テキスト生成だけなら即日実装可能 | 未着手 |
| **4** | フェーズ3: リーガル | チェックリスト自動化 | 未着手 |
| **5** | フェーズ2: 実装自動化 | 最も難易度が高い。品質担保が課題 | 未着手 |
| **6** | 毎日実行パイプライン | 全スクリプトを cron/scheduled agent で結合 | 未着手 |

---

## スクリプト設計方針

毎日自動実行に向けて、各スクリプトは以下を守る:

1. **対話的な入力を求めない**（CLI引数 or 設定ファイルで完結）
2. **冪等にする**（2回実行しても壊れない）

```
365pj/
├── scripts/
│   ├── init-app.ts        ← フェーズ1: アイデア→アプリ初期化
│   ├── deploy.sh          ← フェーズ4: ワンコマンドデプロイ
│   ├── legal-check.ts     ← フェーズ3: リーガルチェック
│   ├── generate-pr.ts     ← フェーズ5: マーケ文生成
│   └── analytics.ts       ← フェーズ5: 反応トラッキング
├── .claude/
│   └── commands/
│       ├── new-app.md     ← /new-app でアプリ初期化
│       ├── ship.md        ← /ship でデプロイ+PR生成
│       └── legal.md       ← /legal でリーガルチェック
```

---

## 「全自動」と「半自動」の線引き

- **全自動**: ファイルコピー、環境変数設定、デプロイ、OGP生成、ログ記録
- **半自動（生成→確認→実行）**: SNS投稿、リーガル判断、アプリ名決定、設計の最終判断
- **手動のまま**: アイデア選定の最終判断、課金モデルの決定

> 「考える時間」は人間に残して「作業時間」を0にする。
